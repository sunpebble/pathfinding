# Cloudflare Workers — One-Time Setup Runbook

Audience: operator with write access to the Sunpebble Cloudflare account and the
`sunpebble/pathfinding` GitHub repo.

Scope: everything that must be done **once** before the API can serve traffic on
`https://api.trips.sunpebblelabs.com`. After this runbook the only remaining
first-deploy step is the CI workflow in Task 6.4 + the smoke checks in Task 7.1.

Source of truth for bindings: [`packages/api/wrangler.jsonc`](../../../packages/api/wrangler.jsonc).
Build output (generated by `flue build --target cloudflare`) lives at
`packages/api/dist/<worker>/wrangler.json`, where `<worker>` is the Flue-emitted
worker directory name (resolve it with `ls -d packages/api/dist/*/`).

> Do **not** run any `wrangler` command from inside this session — the commands
> below are meant to be copy-pasted into the operator's own shell where
> `npx wrangler login` has completed.

## 1. Create the D1 database

Run from `packages/api/` so wrangler reads `wrangler.jsonc` automatically:

```bash
npx wrangler d1 create pathfinding
```

Expected output:

```
✅ Successfully created DB 'pathfinding'
[[d1_databases]]
binding = "DB"
database_name = "pathfinding"
database_id = "<UUID-like string>"
```

**Backfill `database_id` into [`packages/api/wrangler.jsonc`](../../../packages/api/wrangler.jsonc):**

Replace the literal `REPLACE_AFTER_wrangler_d1_create` with the printed UUID.

```jsonc
"d1_databases": [
  {
    "binding": "DB",
    "database_name": "pathfinding",
    "database_id": "<paste UUID here>",
    "migrations_dir": "../../packages/database/drizzle"
  }
]
```

Verify the file parses (still valid JSONC) and the placeholder is gone:

```bash
grep -n REPLACE_AFTER packages/api/wrangler.jsonc   # expect: no match
```

> ⚠️ Without a real `database_id`, every subsequent `wrangler d1 …` / `deploy`
> will fail with "database not found". This is the single most common setup error.

## 2. Create the R2 bucket

```bash
npx wrangler r2 bucket create pathfinding-uploads
```

Expected: `Created bucket 'pathfinding-uploads'`.

The bucket name is wired to the `UPLOADS` binding in `wrangler.jsonc` — no
backfill needed (R2 buckets are addressed by name, not by UUID). Verify:

```bash
npx wrangler r2 bucket list | grep pathfinding-uploads
```

## 3. Apply D1 migrations (first time)

Migration SQL lives in [`packages/database/drizzle/`](../../../packages/database/drizzle)
(`0000_yummy_robin_chapel.sql` = schema, `0001_updated_at_triggers.sql` = the
SQLite `AFTER UPDATE` triggers that emulate MySQL's `ON UPDATE NOW()`). The
`migrations_dir` is already declared in `wrangler.jsonc`, so **no `--migrations-dir`
flag is needed** — wrangler resolves it from config.

Two equivalent paths:

**Option A — apply from source config (recommended for one-time setup, no build needed):**

```bash
cd packages/api
npx wrangler d1 migrations apply pathfinding --remote
```

Wrangler reads `wrangler.jsonc` (incl. the `database_id` you just backfilled and
`migrations_dir`), finds the two pending migration files, and applies them to the
remote D1 in order. Confirm with `y` at the prompt.

**Option B — apply from built dist config (what CI will do in Task 6.4):**

```bash
pnpm --filter @pathfinding/api exec flue build --target cloudflare
DIST=$(ls -d packages/api/dist/*/ | head -n1)   # trailing slash included
npx wrangler d1 migrations apply pathfinding --remote --config "${DIST}wrangler.json"
```

Option B requires a successful `flue build` first (Task 6.3). For one-time setup
prefer **Option A** — it touches the same DB with the same SQL.

Verify both migrations landed:

```bash
npx wrangler d1 execute pathfinding --remote \
  --command "SELECT name FROM d1_migrations ORDER BY id"
# expect: 0000_yummy_robin_chapel, 0001_updated_at_triggers
```

## 4. Set secrets (persist across deploys)

Secrets are stored server-side, keyed by worker name (`pathfinding-api`). They
survive every subsequent deploy and never appear in the repo. Set them once:

```bash
cd packages/api
npx wrangler secret put JWT_SECRET          # paste prod JWT signing secret, Enter
npx wrangler secret put DEEPSEEK_API_KEY    # paste DeepSeek key, Enter (only if AI chat is enabled)
```

Each prompts on stdin — paste the value, press Enter. Wrangler prints
`Success! Uploaded secret JWT_SECRET`. Values are not echoed and not stored locally.

> ℹ️ Order-independent: secrets can be set **before or after** the first deploy.
> If set before, they attach to the script on first deploy; if set after, they
> apply immediately with no redeploy required.

Any other env value that must stay out of git (future provider keys, etc.) goes
through the same `wrangler secret put <NAME>` flow — add it to this list when
introduced.

## 5. Confirm / edit `vars` (non-secret env)

"vars": { "CORS_ORIGIN": "https://trips.sunpebblelabs.com" }

Before first deploy, confirm:

- `CORS_ORIGIN` matches the dashboard's real public origin. If the dashboard is

  Troubleshooting below.
- Add any additional non-secret vars (`ADMIN_EMAILS`, log level, etc.) here as
  needed; secrets must never go in `vars`.

## 6. CI API token + GitHub repo secrets

The deploy workflow (Task 6.4) authenticates to Cloudflare with a scoped API
token stored in GitHub. Create it once:

### 6.1 Create the Cloudflare API token

1. Cloudflare dashboard → **My Profile → API Tokens → Create Token → Create Custom Token**.
2. Permissions (exactly these — least privilege):

   | Resource                         | Permission |
   | -------------------------------- | ---------- |
   | **Account · Workers Scripts**    | Edit       |
   | **Account · D1**                 | Edit       |
   | **Account · Workers R2 Storage** | Edit       |
   | **Account · Account Settings**   | Read       |

3. Account Resources: Include — the Sunpebble account.
4. (Zone Resources not required — Custom Domain lives under Workers Scripts.)
5. Create Token, copy the token value immediately (shown once).

Also grab the **Account ID**: dashboard → any domain → right sidebar →
"Account ID" (or `npx wrangler whoami` after `wrangler login`).

### 6.2 Add GitHub repo secrets

Repo → **Settings → Secrets and variables → Actions → New repository secret**:

| Secret name             | Value               |
| ----------------------- | ------------------- |
| `CLOUDFLARE_API_TOKEN`  | token from 6.1      |
| `CLOUDFLARE_ACCOUNT_ID` | account ID from 6.1 |

These are referenced by the deploy workflow's `env:` blocks. They never need
rotation unless the token is revoked.

## 8. Smoke / verification (after first deploy)

Run once the Task 7.1 deploy is green:

```bash
# DNS: expect an A record (Workers custom domain publishes one automatically)
dig +short api.trips.sunpebblelabs.com A

# Health: expect HTTP 200 and a JSON body
curl -sS -m 15 -w "\nHTTP %{http_code}\n" https://api.trips.sunpebblelabs.com/health

# CORS preflight: expect 200/204 and an Access-Control-Allow-Origin header
curl -sS -m 15 -o /dev/null -D - -X OPTIONS \
  -H "Origin: https://trips.sunpebblelabs.com" \
  -H "Access-Control-Request-Method: GET" \
  https://api.trips.sunpebblelabs.com/health
```

Then a functional smoke (signup → signin → authenticated call):

```bash
BASE=https://api.trips.sunpebblelabs.com
curl -sS -X POST "$BASE/api/auth/signup" -H 'Content-Type: application/json' \
  -d '{"email":"ops+smoke@sunpebblelabs.com","password":"'"$SMOKE_PWD"'"}'
# …then /api/auth/signin for a token, then GET /api/itineraries with the Bearer token
```

## 9. Rollback

Rollback re-points the `api.trips.sunpebblelabs.com` Custom Domain at the previous
Worker deployment. **The Custom Domain itself does not change** — no DNS wait,
certificate stays valid, rollback is effectively immediate.

The `--config` target must be the **built** wrangler config (needs a prior
`flue build`). On the operator's machine:

```bash
cd packages/api
pnpm --filter @pathfinding/api exec flue build --target cloudflare   # if dist/ is stale
DIST=$(ls -d dist/*/ | head -n1)

# 1. List recent deployments and copy the target version/point
npx wrangler deployments list --config "${DIST}wrangler.json"

# 2. Roll back to the previous deployment
npx wrangler deployments rollback --config "${DIST}wrangler.json"
```

`rollback` with no extra args reverts to the immediately previous deployment; to
target a specific one, pass `--message`/version per `wrangler deployments rollback --help`.

Notes:

- Rollback does **not** revert D1 schema. If a bad migration shipped, you must
  write a forward-fix migration (D1 has no `down`).
- Secrets and vars are unaffected by rollback — they're independent of code versions.
- In CI, the same two commands can be added to a `workflow_dispatch` rollback job.

## 10. Troubleshooting

| Symptom                                                                                                                  | Cause                                                     | Fix                                                                                                                                                                             |
| ------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `database not found` / `d1: database_id is required`                                                                     | `database_id` still `REPLACE_AFTER_…` in `wrangler.jsonc` | Re-run Step 1, backfill the UUID, re-run the failing command                                                                                                                    |
| `wrangler d1 migrations apply` says "No migrations to apply" but tables missing                                          | Wrong `migrations_dir` or ran from wrong cwd              | Confirm `migrations_dir: "../../packages/database/drizzle"` and that you're in `packages/api/`; or pass `--config` to the built dist config                                     |
| `/health` 200 but every CORS request fails in browser                                                                    | `CORS_ORIGIN` empty or mismatched                         | A missing/empty `CORS_ORIGIN` makes the API **wide-open** (or rejected, depending on middleware) — set it to the exact dashboard origin in `wrangler.jsonc` `vars` and redeploy |
| Deploy error: `Some functionality requires compatibility flag 'nodejs_compat'` or runtime crash on `Buffer`/`scryptSync` | `nodejs_compat` flag missing from config                  | Confirm `compatibility_flags: ["nodejs_compat"]` in `wrangler.jsonc` (it is, by default) — if a custom build drops it, restore it                                               |
| `wrangler secret put` → `Publishing service limited to an existing script`                                               | Worker not yet deployed (older wrangler builds)           | Either deploy once first (Task 7.1), or upgrade wrangler (`pnpm update @flue/cli`) — current wrangler accepts secrets pre-deploy                                                |
| `flue build` emits no `dist/<worker>/wrangler.json`                                                                      | Build failed silently / wrong target                      | Re-run `pnpm --filter @pathfinding/api exec flue build --target cloudflare` and read full output; the worker dir is the emitted top-level entry under `dist/`                   |
| Custom Domain stuck in `Initializing`                                                                                    | DNS propagation or cert issuance                          | Cloudflare provisions DNS + cert automatically; wait ~1–5 min; verify with `dig +short api.trips.sunpebblelabs.com`                                                             |
| `wrangler deployments rollback` errors                                                                                   | No previous deployment exists (first deploy)              | A rollback requires ≥2 deployments; on a brand-new worker, push a fix forward instead                                                                                           |

## 11. Idempotency / re-runs

- `wrangler d1 create pathfinding` is **not** idempotent — re-running errors with
  "database exists". To recreate, first `wrangler d1 delete pathfinding` (destroys data).
- `wrangler r2 bucket create` is idempotent-safe to retry only if the bucket is
  yours and empty; on existing non-empty bucket it errors. Use `r2 bucket list` to check.
- `wrangler d1 migrations apply` is idempotent — it skips already-applied migrations
  via the `d1_migrations` table.
- `wrangler secret put` is idempotent — it overwrites the named secret.
